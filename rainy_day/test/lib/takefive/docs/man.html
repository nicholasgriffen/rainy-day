Take five!

A semantically-consistent pure-CSS slide renderer

Manual

Introduction

Take five! is a slide renderer written in pure CSS and oriented to the semantics of HTML (semantic nodes, microdata, microformats, RDFa). It can be used as a simple lightbox for images and videos, or as a complete slide viewer for complex contents. Before further reading, please consider to see the live demo.

Getting started

In order to create a slide with Take five! all you need to do is to include takefive.css in your document, then use one of the following basic syntaxes:

  • <article id="my_hash" class="slide">
        <element-to-be-slided />
    </article>
  • <article id="my_hash" itemscope class="slide">
        <nav>
            <a href="#nowhere" rel="parent"></a>
        </nav>
        <element-to-be-slided />
    </article>
  • <article id="my_hash" itemscope class="slide">
        <element-to-be-slided />
        <nav>
            <a href="#nowhere" rel="parent">My diary</a>
        </nav>
    </article>
  • <article id="my_hash" itemscope class="slide">
        <nav>
            <a href="#nowhere" rel="parent">My home page</a>
            <a href="#my_hash" rel="prev">[previous slide name]</a>
            <a href="#my_hash" rel="next">[next slide name]</a>
        </nav>
        <header>
            <h1>[title]</h1>
        </header>
        <this-element-will-be-slided />
        <this-element-will-be-displayed-on-the-right-side-of-the-screen />
        <this-element-will-be-displayed-on-the-right-side-of-the-screen />
        <etc. />
        <footer>[footer]</footer>
        <aside>[aside]</aside>
    </article>
  • etc.

Example

The following minimalistic code

<p style="text-align: center; font-size: 2em;">
    <a href="#photo_of_the_day">Photo of the day</a>
</p>

<article class="slide" id="photo_of_the_day" itemscope itemtype="http://schema.org/ImageObject">
    <nav><a href="#nowhere" rel="parent">My diary</a></nav>
    <figure>
        <img src="images/1.jpg" itemprop="image" alt="Photo of the day" />
        <figcaption itemprop="caption">Photo of the day</figcaption>
    </figure>
</article>

will produce this result.

Slides containers (viewports)

Each slide is defined through the element <article id="my-hash-tag" class="slide"> … </article>. Such element is constructed so that:

  • it is independent and does not require any special container; it can nevertheless link to other slides, forming so a gallery
  • it can contain any kind of HTML elements, in any number and in any order (but be aware of the limits of the screen)
  • no further classes are required by the CSS, beside the "slide" class defined above (but a few additional classes are available for special options)
  • it is compatible with microdata, microformats and RDFa.

Several slides belonging to a single gallery can be logically packed together within any parent element – normally a <section> element is chosen for this.

Each slide is displayed fixed, fullscreen and can be accessed through its hashtag.

Children elements

Direct children of the viewport

  1. The first <nav> element found to be direct child of the slide will be placed on the left side of the viewport. Other <nav> elements found to be direct children of the slide will be normally displayed within the flow body of the main parent (i.e., on the right side, see below).
  2. The first <header> element found to be direct child of the slide will be placed on the top side of the viewport. Other <header> elements found to be direct children of the slide will be normally displayed within the flow body of the main parent (i.e., on the right side, see below) – but consider to use only one <header> element for each scoped parent.
  3. The first <footer> element found to be direct child of the slide will be placed on the right-bottom corner of the viewport. Other <footer> elements found to be direct children of the slide will be normally displayed within the flow body of the main parent (i.e., on the right side, see below) – but consider to use only one <footer> element for each scoped parent.
  4. The first <aside> element found to be direct child of the slide will be placed on the left-bottom corner of the viewport. Other <aside> elements found to be direct children of the slide will be normally displayed within the flow body of the main parent (i.e., on the right side, see below).
  5. The first direct child found to be neither <nav>, nor <header>, nor <footer>, nor <aside> will be placed at the center of the slide and considered to be the main content of the slide. From now on, this will be called “The Fifth Element” or “the <leeloo> element” (after Luc Besson's movie).
  6. Further direct children that do not match the previous conditions will be displayed on the right side of the screen, in their normal order; the overflow property of this portion of the slide is set to hidden.

All the elements listed above, with the exception of the <leeloo> element, are optional and can be omitted. Note also that their order is free.

Non-direct children of the viewport (sub-children)

Sub-children of the viewport will be displayed normally, in their natural order. The only exception regards link elements that are children of the main <nav>, which will be considered as internal navigation links and styled consequently (see below).

Children of the main <nav> (the rel attribute)

Within the main <nav> container – and in one case within the <leeloo> element (see below) – Take five! is sensitive to the rel attribute. In particular:

  1. article.slide > nav:first-of-type a[rel~="index"] will be accompained by the label “Index”
  2. article.slide > nav:first-of-type a[rel~="first"] will be accompained by the label “First slide”
  3. article.slide > nav:first-of-type a[rel~="next"] will be accompained by the label “Next slide”
  4. article.slide > nav:first-of-type a[rel~="prev"] will be accompained by the label “Previous slide”
  5. article.slide > nav:first-of-type a[rel~="last"] will be accompained by the label “Last slide”
  6. article.slide > nav:first-of-type a[rel~="bookmark"] will be accompained by the label “Share it!”
  7. article.slide > nav:first-of-type a[rel~="author"] will be accompained by the label “Author”
  8. article.slide > nav:first-of-type a[rel~="tag"] will be accompained by the label “Tagged content”
  9. article.slide > nav:first-of-type a[rel~="child"] will be accompained by the label “More” and styled differently
  10. article.slide > nav:first-of-type a[rel~="parent"], will be showed in background and accompanied by the label “Back to” when not empty, or by the label “Click to exit” when empty
  11. leeloo a[rel~="parent"]:empty will entirely cover the <leeeloo> element

Each of the elements matching the selectors 1-11 above may be omitted (but either 10 or 11 should be present in order to give the possibility to exit from the slide), and all of them can cohabitate together (note: also elements 10 and 11).

Other a[rel="…"] elements not listed above, but placed inside the main <nav>, will be accompanied by a label containing their rel attribute. Tag links placed anywhere else within the slide (i.e., article.slide a[rel="tag"]) will be styled with a star preceding the text.

Custom labels can express a pseudo-relationship in links without a rel attribute, or replace default labels in links with a rel attribute (tags' stars as well), by setting a data-label attribute on the link – as in <a href="#about" data-label="See also">About</a> or <a href="#my-sweetheart" rel="sweetheart" data-label="Thanks to">Carla</a>.

Furthermore, it is possible to remove the default labels by assigning a "no-label" class to the link – as in: <a href="#sample-photo" rel="next" class="no-label">Sample photo</a>.

Elements matching the following selectors will generate navigation buttons.

  1. article.slide > nav:first-of-type a[rel~="parent"]:not(.no-nav) generates the × button
  2. article.slide > nav:first-of-type a[rel~="next"]:not(.no-nav) generates the ❯ button
  3. article.slide > nav:first-of-type a[rel~="prev"]:not(.no-nav) generates the ❮ button
  4. article.slide > nav:first-of-type a[rel~="index"]:not(.no-nav) generates the ⌂ button
  5. article.slide > nav:first-of-type a[rel~="last"]:not(.no-nav) generates the ⇥ button
  6. article.slide > nav:first-of-type a[rel~="first"]:not(.no-nav) generates the ⇤ button
  7. article.slide > nav:first-of-type a[rel~="bookmark"]:not(.no-nav) generates the 🔗 button
  8. article.slide > nav:first-of-type a[rel~="author"]:not(.no-nav) generates the @ button

When present, it is very important that each of them occur only once for each slide, or the button will be generated twice in the same position. Setting a "no-nav" class on the links will prevent the creation of buttons and will therefore allow duplications – a typical case is the duplication of a[rel~="author"] in slides with more than one author (but consider to use a dedicated slide for listing more than one author). Other a[rel="…"] elements (for example, a[rel="archives"], or a[rel="child"], or a[rel="tag"]), do not generate buttons and may occur more the once within the main <nav>. Moreover, additional duplications of the above cited a[rel="…"] elements, but placed elsewhere (e.g.: article.slide > div a[rel="next"]), are irrelevant for this CSS.

The a[rel~="parent"] element

In order to exit from the viewport the a[rel~="parent"] element is used (see above). Normally this links to any unexisting hash (in our examples we used the hash "#nowhere", but a simple "#" can be enough), but in the case of nested galleries it can optionally link to the parent gallery (see below).

The special link a[rel~="parent"] complies with the following rule:

  • If it is inside the main <nav> ⇒ it will be shown on the right-top corner; a transparent, clickable, full-screen layer will be created.
  • If it is inside the <leeloo> element and is empty ⇒ a transparent clickable layer will cover the whole (and only) <leeloo> element.
  • If, instead,
    • it is inside the <leeloo> element but is not empty,
    • or it is outside both the main <nav> and the <leeloo> element (empty or not empty),
    • or a[rel~="parent"] *is* the <leeloo> element iself
    then ⇒ it will be ignored by this CSS and displayed as a normal link.
More than one token within the rel attribute

It is syntactically and semantically valid to assign more than one token to the rel attribute. As explained in the W3C Recommendation 28 October 2014,

The attribute's value must be a set of space-separated tokens.

In this case, within the main <nav>, the CSS will choose the most important token according to the following order (the more down in the list the less important):

  1. "parent"
  2. "child"
  3. "next"
  4. "prev"
  5. "index"
  6. "last"
  7. "first"
  8. "bookmark"
  9. "author"
  10. "tag"

For instance, <a href="#hello-world" rel="index prev">Hello world</a> will be styled as if it contained just the token "prev", while <a href="#hello-world" rel="prev parent">Hello world</a> will be styled as if it contained just the token "parent", and so on. The token "child" used together with one of the tokens 3-9 listed above will preserve the button generated by the latter.

The possible values for the rel attribute are several. Furthermore:

Extensions to the predefined set of link types may be registered in the microformats wiki existing-rel-values page.

Nested galleries

Galleries can be nested within other galleries, possibly through the <section> element. In this case the link to the child gallery is usually given through a a[rel~="child"] link.

<section>
    <article id="item-1" class="slide">
        <nav>
            <a href="#item-1-1" rel="child">Nested gallery</a>
            <a href="#nowhere" rel="parent">My diary</a>
        </nav>
        <element-to-be-slided />
    </article>
    <section>
        <article id="item-1-1" class="slide">
            <nav>
                <a href="#item-1" rel="parent">Item #1</a>
            </nav>
            <element-to-be-slided />
        </article>
    </section>
</section>

The code above will produce a result similar to this example.

Issues

For compatibility issues with other CSS, here follows the table of all the z-indices assigned by Take five!. They cover the range 11109-11118, therefore no other elements within the document should have a z-index higher than 11109 in order to not be displayed over the viewport.

Targetz-index
article.slide 11109
article.slide > header:first-of-type 11110
article.slide > nav:first-of-type a[rel~="parent"] 11111
article.slide > aside:first-of-type 11112
article.slide > footer:first-of-type 11113
article.slide > *:not(nav:first-of-type,header:first-of-type,footer:first-of-type,aside:first-of-type,leeloo) 11114
leeloo 11115
article.slide > nav:first-of-type a:not([rel~="parent"]) 11116
article.slide > nav:first-of-type a[rel~="parent"]::after 11117
leeloo a[rel~="parent"]:empty::before 11118

Notes

  • Take five! focuses on the semantics aspects of HTML5 (i.e., how a slide can be naturally declared), while the graphical result can still be improved. If you have new ideas, please do not hesitate to propose them on GitHub. Normally I accept good ideas, but in case of proposals too specific for a general-purpose slide renderer, the project can eventually be forked and released again as new project under the terms of the GPL3 (let me know!).
  • For learning more about the itemscope attribute and microdata, please see schema.org. A lists of the main rel attributes can be found here, while its extension, together with information about microformats, can be found here.
  • Take five! aims to be compatible with microdata, microformats and RDFa. But neither microdata, nor microformats, nor RDFa are required by Take five!: the only required condition is a class="slide" assigned to an <article> element.
  • It is possible to prevent the hashtag from being displayed in the viewport by using the class "no-hashtag" – as in <article class="slide no-hashtag" id="my-hash-name"> … </article>.
  • It is possible to prevent the creation of navigation buttons by main <nav>'s links by assigning the class "no-nav" to them – as in <a href="#sample-slide" rel="next" class="no-nav">Sample slide</a> or <a href="#sample-slide" rel="next" class="no-label no-nav">Sample slide</a>.
  • The opposite, i.e. generating the navigation buttons without showing the links they refer to, can be obtained by leaving empty the content of the link – as in <a href="#sample-slide" rel="prev"></a>.
  • The slide container (article.slide) can optionally possess a data-context attribute, whose content will be displayed on the left-bottom of the viewport – as in <article class="slide" data-context="Hello world!" id="my-hash-name"> … </article>.
  • Take five! does not require JavaScript, but it does not discourage its usage. JavaScript can be useful in many ways, such as touch events for navigating through the slides, printing of single contents, etc.
  • Feel free to make experiments. Take five! is very flexible, but in in case of problems, please do not hesitate to write.

FAQ

I have many images in my page. How can I avoid that the browser preloads all of them?
If you are interested in this slide renderer, probably you don't want to leave any semantic content out of your HTML (as it would be to load the slides through JavaScript, for example). But if you really need to, consider to use the <template> element.
How can I enable the escape key for closing the slides?
You need to use JavaScript. I intentionally did not write any line of JavaScript code for this CSS, but you can work around it. Here follows a basic script for closing the slides by pressing the escape key.
/* closes the slide currently open or does nothing */
function closeCurrentSlide () {
    var oSlide = document.querySelector("article.slide:target");
    if (oSlide) {
        var oExitLink = oSlide.querySelector("nav:first-of-type a[href][rel~=\"parent\"]");
        location.assign(oExitLink ? oExitLink.href : "#");
    }
}

document.addEventListener("keyup", function (oEvt) {
    if (oEvt.key === "Escape") {
        closeCurrentSlide();
    }
}, false);
How will the slides look on a printed page?
They won't, probably. Consider to use Take five! only for the view on screen, then write a custom CSS for the printed version of your page:
<link rel="stylesheet" type="text/css" href="takefive_min.css" media="screen" />
<link rel="stylesheet" type="text/css" href="print.css" media="print" />
A good starting point for your print.css might be:
a {
    color: #000000;
    text-decoration: none;
}

a[href^="http"]::after {
    content: "(" attr(href) ")";
    vertical-align: super;
    font-size: x-small;
}

nav, aside {
    display: none;
}

article.slide {
    page-break-before: always;
    page-break-after: always;
    page-break-inside: avoid;
    background-color: #ffffff;
    color: #000000;
    font-family: inherit;
    text-align: justify;
}
